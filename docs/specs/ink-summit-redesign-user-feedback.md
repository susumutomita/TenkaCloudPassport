# Ink / Summit リデザイン ユーザーフィードバック

## 対象と役割

- Issue: <https://github.com/susumutomita/TenkaCloudPassport/issues/70>。
- Role: 次の 3 つのペルソナによる机上 Simulation。
  - (a) 対面イベントで初めて Passport を使う参加者。
  - (b) TenkaCloud 本体を日常的に使っている既存ユーザー。
  - (c) 会場で QR を見せ合う Host。
- 参照した正本: [仕様書](./2026-07-20-ink-summit-redesign.md)、[設計](../design/2026-07-20-ink-summit-redesign.md)、[QR 招待・共有確認・Ready フローの設計](../design/qr-invite-and-ready-flow.md)、claude.ai/design「TenkaCloudPassport デザイン見直し」の `TenkaCloud Passport Redesign.dc.html`（デザインデータとして参照）。
- Final verdict: `ALLOW`（リデザインは体験を損なわない。ただし第 2 節の要修正 3 点を実装時に解消すること）。
- 実機・実会場での検証: `Not run`（本書は机上 Simulation であり、Field Ready の証拠には使わない）。

## 1. 旧デザインから Ink / Summit へ変わることの体験影響

### 会場の照明での視認性

対面イベントの会場は、スポットライト・逆光・自動輝度で暗くなった画面という悪条件が重なる。旧デザイン（クリーム #f4f1ea 地 + ink #17201d）に対し、新デザインは白地 #ffffff + ink #1d1d1f で本文コントラストが 16.9:1 まで上がる。明るい会場では光沢画面の映り込みが暗色 UI を潰すため、地を明るく文字を濃くする方向は会場向きの正しい選択であり、初見の参加者（ペルソナ a）が案内文を読み違えるリスクは旧デザインより下がる。

一方で、フォレストグリーンの primary ボタンが持っていた「押す場所が色で分かる」手掛かりは消える。新デザインでは ink 塗りボタンが画面内で最も濃い面になることで代替されており、1 画面 1 primary の構成（デザインデータの 5 画面すべてで primary は 1 つ）を守る限り迷いは生じない。この「1 画面に ink 塗りボタンは 1 つ」という前提は意匠調整の暗黙条件なので、実装時に崩さないでほしい。

### 墨モノクロ UI で状態色ドットは十分目立つか

目立ち方はむしろ向上する。旧デザインは緑（primary #185c4b・primarySoft #dcebe4）が UI 全域に使われており、状態を示す色が装飾の緑に埋もれる構造だった。新デザインでは chroma を持つ要素が状態ドット（success #2f9e63 / warning #c98a14 / info #3b82f6）と summit accent にほぼ限定されるため、モノクロ地の中で色 = 状態という視線誘導が成立する。これはモノクロ化の実利であり、単なる意匠変更ではない。

ただしドット単体は 7px と小さく、腕を伸ばした距離や会場の強い照明下では色の識別に頼れない。デザインデータではすべてのドットに mono ラベル（「Ready」「残り 12 分で期限切れ」「discovering」）が併記されており、この併記が実質の正本だ。第 2 節で述べるとおり、併記を偶然ではなく契約にしてほしい。

### QR 表示画面の白地はスキャンに有利か

有利だ。QR の module は ink #1d1d1f、地とカードと画面背景がすべて #ffffff で、カメラから見た明暗コントラストは実質最大になる。旧デザインのクリーム地 #f4f1ea 上に QR カードを置く構成と比べ、quiet zone 周辺の地色ムラが消える点も読み取り安定に効く（[QR エンコーダサルベージのユーザーフィードバック](./qr-encoder-salvage-user-feedback.md) 第 2 節で挙げた「off-white な余白に弱いリーダー」への予防になる）。Host（ペルソナ c）として、QR を見せる場面で最も信頼できる配色だと評価する。

注意点は 2 つある。第一に、この利点は M3 の renderer が QR カード内の白地と quiet zone を theme の `background` トークンに追従させないことで初めて守られる。将来ダークモードを導入した場合も QR カードだけは白地固定とする必要があり、この制約はどこかに明文で残すべきだ。第二に、画面輝度が自動で下がると白地の利点が消えるため、QR 表示中の輝度確保は引き続き M3 の表示側課題だ（既出の Known follow-ups と同一であり、本 Issue のスコープ外で正しい）。

## 2. 使いにくい・わかりにくいになりうる点

### 要修正 1: disabled ボタンがほぼ見えなくなる（設計書の 3:1 という記述は数値が成立していない）

設計書のエッジケース節は「ink ボタンの disabled は #f5f5f7 地 + disabled（#c7c7cc）文字で 3:1 の UI コンポーネント境界を保つ」とするが、#c7c7cc 文字と #f5f5f7 地のコントラストは約 1.6:1、#f5f5f7 地と白い画面地の境界は約 1.05:1 で、3:1 はどの組でも成立しない。WCAG 上 disabled 要素はコントラスト要件の適用外なので違反ではないが、ユーザー体験としては別問題だ。

具体的に困る場面は Host Invite 画面だ。Host は「自分も Ready にする」を押した後、相手が Ready になるまで待つ。このとき次操作のボタンが disabled で白地にほぼ溶けると、初見の参加者（ペルソナ a）は「次に押すものが存在するのか」を画面から読み取れず、進行が止まったと誤解する。会場は互いに画面を見せ合いながら操作を教え合う場であり、「そこにあるが今は押せない」ことが 1 m 先からでも分かる必要がある。

要望: disabled は #f5f5f7 地 + border #d2d2d7 の枠 + muted #6e6e73 文字程度まで存在感を残し、可能なら「相手が Ready になると押せます」のような理由文言を併設してほしい（文言変更がスコープ外なら、既存文言の範囲で状態が分かる構成にする）。あわせて設計書の 3:1 という記述は実測値で訂正すべきだ。

### 要修正 2: Ready ラベルの文字色がデザインデータのままでは AA を割る

デザインデータの Host Invite 画面は「Ready」ラベルの文字を success #2f9e63（白地で 3.1:1）で描いている。11px 相当の小さい文字であり、AA の 4.5:1 を満たさない。設計書はこのために successText #1f7a49（4.6:1）を新設しており対策自体は正しいが、仕様書の受け入れ基準「5 画面は意匠まで合わせる」とデザインデータの生値が矛盾する。実装者がデザインデータの hex を忠実に写すと AA 違反が再現するため、「意匠とコントラスト契約が衝突したら契約を優先する（ドットは #2f9e63 のまま、文字は successText）」という優先順位を仕様書側にも明記してほしい。設計書は Bridge カードの補助文字（rgba .5 → .68）で同じ判断を既にしており、それを Ready ラベルにも一般化するだけだ。

同様に「未 Ready」ラベルの #86868b（3.5:1）も小さい文字としては不足する。未 Ready は「まだ済んでいない」ことを伝える能動的な情報であり、薄くて読めないと Host が相手の状態を確認できない。muted #6e6e73（5.3:1）への引き上げを要望する。

### 要修正 3: mutedLight の小サイズ本文への流用

デザインデータには mutedLight #86868b を重要情報に使っている箇所がある。「3 / 12 ・機密情報を入力しないでください」（10.5px）と、Owner Question のメモ欄プレースホルダー「空のままでも回答できます」（14px）だ。前者はプライバシー上の注意喚起であり、3.5:1 の小さい文字で流してよい情報ではない。設計書自身が「mutedLight は小さい本文には使わない」と定めているので、この 2 箇所は実装時に muted 以上へ引き上げ、mutedLight は装飾的な eyebrow ラベルと大型 mono ラベルに限定するという設計書のルールを機械的に守ってほしい。

### 色弱ユーザーで Ready / 未 Ready の判別は成立するか

成立する。ただし条件付きだ。2 型色覚では success #2f9e63 と warning #c98a14 の色相差はほぼ失われ、未 Ready の #c7c7cc との差は主に明度差になる。7px ドットの明度差だけで判別させるのは無理があるが、デザインデータは全状態に文字ラベル（Ready / 未 Ready / discovering / 残り n 分で期限切れ）を併記しており、文字を一次情報・ドットを二次情報とする構成なら色覚に依存しない。

懸念は、この「ドットには必ず文字が付く」がデザインデータ上の偶然にとどまり、仕様書の受け入れ基準にも既存テストにも現れていないことだ。今後の画面追加でドット単独の状態表示が混入すると、そこだけ色覚依存になる。要望として、「状態ドットを文字ラベルなしで単独使用しない」を受け入れ基準またはソース契約テスト（状態ドットのコンポーネント化 + ラベル必須 prop）として固定してほしい。

### そのほかの小さな懸念

- 絵文字選択・チェックボックスの選択状態が「1.5px の ink 枠 + #f5f5f7 地」対「1px の #d2d2d7 枠 + 白地」の差だけで表現される。旧デザインの緑塗りより差が繊細で、会場の照明では見分けにくい。チェックボックス（ink 塗り + チェックマーク）は明確なので、絵文字選択にもチェックマーク等の形状差を足せると安心だ。
- discovering ドットの点滅と Pet 絵文字の pulse はデザインデータ由来のアニメーションだ。既存の reduced-motion 契約テスト（`src/screens/active-lounge-reduced-motion.test.ts`）の対象を意匠変更後も維持してほしい。
- ダークモードはスコープ外で妥当だが、夜の懇親会会場で全面白の画面は眩しい。QR 読み取りには白が有利という第 1 節の理由から現時点の判断を支持するが、将来のダークモード検討時に「QR カードだけ白固定」とセットで扱う旨をフォローアップに残す価値がある。

## 3. TenkaCloud 本体ユーザーから見た一貫性の評価

### 達成されている点

既存ユーザー（ペルソナ b）として、ブランド統一の目的はおおむね達成されると評価する。判断材料は色よりもむしろ構造で、墨モノクロ + 限定的な summit orange という配色戦略、山頂マークと「TenkaCloud Passport」ロックアップの常設ヘッダー、mono 大文字の eyebrow ラベルという 3 点が揃うと、本体・ポータルと同じプロダクト群だと画面を見た瞬間に分かる。旧デザインのクリーム + 緑は単体では良い配色だったが、本体と並べたときに「別会社のアプリ」に見える問題があり、それが解消される。LP の査証スタンプ + MRZ を残したまま色だけ summit 系へ寄せる判断も、Passport という製品固有の記号を消さずにブランドへ従属させる正しいバランスだ。

### 差分として認識しておくべき点

- フォントは同梱せずシステムフォントで実現するため、本体（Inter / Noto Sans JP / JetBrains Mono）と Passport（SF Pro / Roboto と platform monospace）で字形は一致しない。色・マーク・レイアウトで「地続き」を作る方針は妥当だが、本体ユーザーが並べて見れば違いには気づく。仕様書のスコープ外節に明記済みであり、将来フォント同梱を検討するなら別 PR + ADR という現在の整理で十分だ。
- 意匠まで合わせるのはデザインが定義する 5 画面だが、アプリには `SettingsScreen`・`BackupExportScreen`・`QrScanScreen`・`PassportSharePreviewScreen` など 5 画面以外も多数ある。トークン差し替えで色は全画面に波及するものの、山頂マーク付きヘッダー・カード意匠は 5 画面限定となり、対面フローの途中（たとえば Guest として QR を読み取る `QrScanScreen`）でヘッダーの有無が切り替わると、初見の参加者には別アプリへ飛んだように見えかねない。少なくとも対面フロー（share-preview → host-invite → guest-scan → guest-share-preview → Lounge → Outcome）で通過する画面群にはロックアップを揃えるか、残画面への展開をフォローアップとして明示してほしい。

## 4. 受け入れ基準のユーザー視点確認

| 受け入れ基準 | ユーザー視点での意味 | 判定 |

| 山頂マークを共通コンポーネント化し 5 画面のヘッダーへ表示する。 | 一目で TenkaCloud 群と分かる。 | 充足見込み。対面フローで通過する 5 画面以外への展開方針を残すこと。 |

## 5. まとめ

リデザインの方向性（白地 + 墨 + 限定 accent、状態色の希少化、QR の白地最大コントラスト）は、会場での視認性・スキャン成功率・ブランド認知のいずれから見ても旧デザインより優れており、ペルソナ 3 者とも移行に賛成する。条件は第 2 節の要修正 3 点、すなわち disabled の存在感確保と設計書の数値訂正、Ready / 未 Ready ラベルの AA 確保（意匠より契約優先の明文化）、mutedLight の重要情報への流用禁止の徹底だ。加えて、状態ドットの文字ラベル必須化の契約化と、対面フロー通過画面へのロックアップ展開方針の明示を要望として残す。
