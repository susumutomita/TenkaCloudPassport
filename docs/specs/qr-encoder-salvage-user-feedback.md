## 1. M3 実カメラ読取シナリオのシミュレーション

M3 では、会場で Host がスマホに Invite QR を表示し、対面の Guest が自分のスマホの実カメラで読み取って Lounge に参加する場面を想定する。読み取りやすさを左右する変数を、エンコーダの実際の容量表（byte mode・誤り訂正 M）に基づいて確認した。

### QR の密度と Version の関係

エンコーダは payload の byte 数に応じて最小の Version を自動選択する。主要な境界は次のとおり。

| Version | 一辺の module 数 | 収容できる byte 数（誤り訂正 M） |
| --- | --- | --- |
| 10 | 57 | 213 |
| 16 | 81 | 450 |
| 17 | 85 | 504 |
| 20 | 97 | 666 |
| 22 | 105 | 779 |
| 26 | 121 | 1,059（wire format 上限の 1,024 byte はここに収まる） |

上限の 1,024 byte を使い切ると Version 26（一辺 121 module、quiet zone 4 module を足すと 129 module）になる。Version 40（177 module）まで行かない設計は歓迎するが、121 module でも幅 330 pt 程度で表示すると 1 module は約 2.5 pt しかなく、対面距離（20〜40 cm）でカメラ側が 1 module あたり十分な画素を確保するには、近づいてピントが合うことと手ブレの少なさの両方が要る。

一方、現行の Lounge Invite（`src/domain/lounge-invite.ts` の schema v2 を `TCPQ1:` envelope で包んだもの）は、Lounge ID・Join Secret・Transport Fingerprint（`sha256_` + 64 桁）・時刻 2 つ・定員・Capability・Discovery Hint で概算 450〜650 byte に収まり、Version 16〜20（81〜97 module）程度で済む。`hostDiscoveryHint` が日本語で最大の 128 文字（UTF-8 で最大 384 byte）に達すると 750 byte 超・Version 22 前後まで膨らむ。つまり、読み取りやすさはエンコーダではなく payload の太り方で決まる。

### 会場の現実条件との突き合わせ

- 照明: 会場のスポットライトや逆光は画面のグレア（映り込み）を生む。誤り訂正 M（約 15 ％を復元）は、部分的なグレアや Host の指がかかる程度なら吸収できる、妥当な中間選択だと感じる。L に下げて密度を稼ぐより M 固定のほうが安心できる。
- 距離: 対面での自然な距離は 20〜40 cm 程度になる。Version 16〜20 ならこの距離で問題ないが、Version 26 では近づく必要があり、古い端末では最短撮影距離を割ってピントが合わなくなる恐れがある。
- 画面輝度: 自動輝度で暗くなった画面はコントラスト不足で読み取りが遅くなる。これはエンコーダではなく M3 の表示側の責務である（第 4 節の要望）。
- 手ブレ: Host も Guest も手持ちであり、両手が同時に揺れる。高密度になるほどデコード成功に必要な静止時間が延びるため、体感の「かざせばすぐ入れる」を守るには Version を低く保つことが効く。

結論として、エンコーダ自体の仕様（最小 Version 自動選択・誤り訂正 M・上限 Version 26）は、payload が小さいほど QR も小さくなるという読取体験側の要請と一致している。ただし「1,024 byte まで入るから安心」ではなく「1,024 byte を使い切ると読み取りが体感で遅くなる」と理解して payload を設計してほしい。

## 2. 使いにくい・わかりにくいになりうる点

- 上限いっぱいの payload は高密度 QR になる。1,024 byte・Version 26 は仕様上は正しく encode できるが、会場の実カメラでは読み取り時間が延び、「かざしているのに反応しない」という第一印象を作りやすい。
- 古い端末・廉価端末のカメラは、低解像度や固定フォーカスのために高密度 QR に弱い。Guest 側の端末は選べないため、Host 側（表示する QR）を小さく保つことでしか救えない。
- 画面対画面の読み取り特有の問題として、モアレや OLED の PWM 調光によるちらつきがカメラ側に縞を作ることがある。誤り訂正 M はある程度吸収するが、表示サイズと輝度の工夫が必要である。
- ダークモードで QR の明暗が反転すると、多くのリーダーは読めない。encoder は boolean matrix を返すだけで色を持たないため、これは M3 renderer の責務だが、受け入れ基準として明文化されていないと事故になる。
- quiet zone（4 module の余白）は現在テスト内の画素化関数にしか存在しない。M3 renderer が余白を忘れると、matrix が正しくても読めない QR になる。設計上「描画・quiet zone は renderer 側の責務」と書かれているのは正しい分離だが、責務の受け渡しが明文化された受け入れ基準になっていない。
- 実カメラは同じ QR を毎フレーム decode し続ける。M1 の protocol 層は同じ生文字列の 2 回目を `DUPLICATE_SCAN` として拒否するため、M3 の scanner adapter が読取成功時にスキャンを止める（debounce する）設計にしないと、正常な読取直後に重複エラーが見えるという分かりにくい体験になり得る。
- jsQR による round-trip テストは理想的な画素（ブレ・歪み・ノイズなし）での検証であり、会場での読み取りやすさの証拠にはならない。M1 設計も実機検証を人間検証待ちとしており、この位置づけを M3 でも維持すべきである。

## 3. ユーザー視点での受け入れ基準の充足確認

仕様書の受け入れ基準を、Host / Guest の体験にとって何を意味するかで確認した。

| 受け入れ基準 | ユーザー視点での意味 | 判定 |
| --- | --- | --- |
| Version 1〜26 の matrix を純関数で生成する。 | payload が小さければ QR も小さくなり、読み取りが速い。最小 Version の自動選択は体験に直結する。 | 充足。 |
| 既知ベクタテストで回帰を防ぐ。 | ある日突然読めない QR が出る事故を防ぐ。 | 充足。 |
| jsQR round-trip が ASCII・日本語 UTF-8・1,024 byte 境界を検証する。 | `hostDiscoveryHint` の日本語が壊れず読めることの機械保証がある。 | 充足（ただし実カメラ品質の証拠ではない点は第 2 節のとおり）。 |
| 1,024 byte 超は `DATA_TOO_LARGE` で fail-closed になる。 | 「表示はされたが誰にも読めない QR」を Guest に見せない。wire format の上限（`QR_PAYLOAD_MAX_BYTES` = 1024）と一致しており、encode できたものは必ず protocol 的にも受理範囲である。 | 充足。 |
| encode が 100 ms 未満で完了する。 | Host が Invite を発行してから QR が表示されるまでの待ちを感じない。 | 充足。 |
| M1 の `QrCodeView` / `qr-matrix.ts` は変更しない（スコープ外）。 | 現在の M1 体験（装飾 Grid・Screenshot で Payload が判読できない保証）が変わらない。サルベージが既存の使い勝手を壊さない。 | 充足。 |

総合すると、今回のサルベージは M3 体験の前提を壊しておらず、むしろ整えている。前提とは「標準準拠でスキャン可能な QR を依存追加なしで payload に応じた最小サイズで作れること」を指す。`ALLOW` とする。

## 4. M3 に向けた要望

1. payload を小さく保つ予算を明文化してほしい。典型的な Lounge Invite の encode 後サイズに上限予算（例: Version 17・504 byte 以下）を置き、回帰テストで固定してほしい。1,024 byte は「protocol が壊れない上限」であって「快適に読める上限」ではない。
2. M3 renderer の受け入れ基準に、quiet zone 4 module 以上・ダークモードでも明暗を反転しない（常に明るい地に暗い module）・表示中の画面輝度を上げる、の 3 点を含めてほしい。encoder が正しくても、この 3 点のどれが欠けても会場では読めない。
3. 読み取り失敗時の再試行導線を用意してほしい。一定時間デコードできないときに Guest 側へ「近づける・明るくする・反射を避ける」のヒントを出し、Host 側には QR の再表示（必要なら Invite の再発行と Secret Rotation）への導線を置いてほしい。
4. 実カメラの連続デコードと `DUPLICATE_SCAN` の関係を整理してほしい。読取成功時にスキャンを止める debounce を scanner adapter 側に持たせ、正常な 1 回の読取が重複エラーとして見えないようにしてほしい。
5. 実機・実会場の人間検証を M3 の完了条件に含めてほしい。少なくとも「古めの Android 端末 1 台・屋内スポットライト直下・対面距離」の組み合わせで、読取成功までの体感時間を確認してほしい。jsQR round-trip の緑は代替にならない。
6. `hostDiscoveryHint` の日本語長文が QR を太らせる主要因になり得るため、Hint の実運用上の推奨長（または encode 後サイズへの寄与の可視化）を M3 の設計時に検討してほしい。
